三项高级能力的定位与关系

M2.5 编排 · v1 跨源聚合 · M2.5+ 自定义源

这三项是 deepferry 区别于"又一个 MCP 数据连接器"的核心。编排引擎解决"如何拿到数据"——把企业平台的多步请求流程（认证→发现→路由）封装成单一数据源； 跨源聚合解决"如何重组数据"——用一条 SQL 把分散在 MySQL、PostgreSQL、登录态 HTTP API 里的数据 JOIN 成一份报表； 自定义数据源是逃生通道——当固定流程混 HTTP+DB或需要声明式引擎拒绝的逻辑（条件/循环/中流聚合）时，用命令式 Python 写成一个 DataSource 类。 三者可叠加：编排源/自定义源产出的数据，正是跨源 JOIN 的一极。Agent 多轮编排（query+params 串联）是 MCP 基线用法，与三项能力正交。

编排引擎 · 内容场景

Orchestration M2.5

企业级数据平台几乎从不直接暴露数据库——它们要求"登录 → 列出账号可见实例 → 对选定实例路由 SQL"。 静态 HTTP 源无法表达这种流程（资源只能配置时枚举），而编排引擎把它声明在 config.toml 里，封装为一个普通的 DataSource。 下面是 4 个它能覆盖的真实业务场景。

1 企业多租户 DB 查询平台 核心场景

login → discover 账号实例 → route SQL。编排引擎的" raison d'être"。

🏷️ 业务背景 · 为什么静态 HTTP 源做不到
某公司内部"统一数据查询平台"不直连数据库，而是提供 HTTP API：先 POST /auth/login 拿 token， 再 GET /instances 列出当前账号有权访问的 DB 实例（每人不同）， 最后 POST /instances/<id>/query 路由 SQL。 实例列表是登录后才知道的，无法在 TOML 里预声明——这正是编排引擎要解决的：mvp-init 的原子 HTTP 源做不到。

声明式配置（config.toml）

# 一个编排源 = 多步流程，对 Agent 是普通数据源
[[sources]]
id = "db-platform"
type = "orchestrated"
base_url = "https://db-platform.internal/api"

[[sources.steps]]
name = "login"
kind = "auth"            # 产出 token，后续步骤复用
method = "POST"
path = "/auth/login"
body_template = { username = "${DB_USER}", password = "${DB_PASS}" }
output = { token = "access_token" }   # 绑定别名 → {{steps.login.token}}

[[sources.steps]]
name = "discover"
kind = "discover"        # 产出动态资源列表（list_resources 走到这里）
method = "GET"
path = "/instances"
headers = { Authorization = "Bearer {{steps.login.token}}" }
output = { instances = "data" }

[[sources.steps]]
name = "query"
kind = "request"
method = "POST"
path = "/instances/{{inst_id}}/query"   # inst_id 来自 query.params 或 discover[0]
headers = { Authorization = "Bearer {{steps.login.token}}" }
body_template = { sql = "{{sql}}" }       # agent 的 SQL 注入到此
output = { rows = "data" }

2 多区域实例聚合查询（foreach merge）

登录 → 发现 N 个区域实例 → 对每个跑同一份 SQL → UNION 成一张表。

🏷️ 业务背景 · 跨区域汇总
全球 8 个区域各部署一套订单库，通过统一平台 API 暴露。分析师要"各区域近 7 天订单总额排行"—— 需要对 8 个实例分别查询，再把结果合并。foreach 的 merge 模式（UNION）天然适配：一次 query() 拿到合并后的单一结果。

foreach 配置（接续上面的 login/discover）

[[sources.steps]]
name = "query_each_region"
foreach = { over = "{{steps.discover.instances}}", as = "inst", mode = "merge" }
method = "POST"
path = "/instances/{{inst.id}}/query"
headers = { Authorization = "Bearer {{steps.login.token}}" }
body_template = { sql = "{{sql}}" }
output = { rows = "data" }
# merge 模式：8 次迭代的结果 UNION 成一个 StructuredResult
# 要求每次迭代列结构一致，否则 FOREACH_SCHEMA_MISMATCH

3 动态权限资源发现

list_resources() 不读 TOML，而是跑流程到 discover 步骤。

新员工入职后权限变化，可见实例随之增减。编排源的 list_resources() 首次调用或 TTL(300s) 过期时， 才运行到 discover 步骤并物化为 Resource 列表；缓存在内存，auth_retry 时失效（权限可能变更）。
隔离契约：discovery 永不执行 query 步骤——不能仅为"列个实例"就跑用户 SQL。这是硬约束。

4 登录态 SaaS 数据提取

Salesforce / Stripe / 内部 CRM 风格的多步取数。

非 DB 平台但同样"先登录再取数"的 SaaS：登录拿 session → 列出业务对象 → 查询某对象。 编排引擎把它们统一为单一数据源，Agent 无需关心"这个 API 要先登录"——deepferry 把认证隐藏在数据源内部。
401 继承：编排流程内的任一步骤遇到 401，复用 two-step-auth 的响应式路径——失效 token → 重登 → 单次重试，并在该步骤下开 auth_retry 子 span。N 个并发查询仍只触发一次登录。

编排引擎 · 能力小结

✅ 能做

  • 线性流程（login → discover → query）
  • foreach 扇出（merge UNION / separate 并排）
  • 步骤间 {{var}} 插值（Jinja 子集，仅变量引用 + 索引）
  • 动态资源发现（TTL 缓存 + auth_retry 失效）
  • 继承两步认证的 token 缓存与 401 重试

❌ 不做（硬上限）

  • 条件分支 if（需逻辑→写专门 DataSource 实现）
  • 循环 while / 轮询
  • 任意 DAG / 多分支并行（那是 Airflow 的活）
  • Jinja 过滤器 | upper、模板继承（配置加载期拒绝）
  • 用户自定义 Python（安全沙箱噩梦）

自定义数据源 · 内容场景

Custom DataSource · M2.5+

当固定流程混 HTTP+DB或需要声明式引擎拒绝的逻辑（条件/循环/中流聚合），把它封装成一个自定义 DataSource 类。 Agent 仍只调一次 query()，拿到一个 StructuredResult；多步、混合 HTTP+DB 的逻辑住在 Python 的 execute() 里。 这是代码级扩展点——新源类型，按模块 review/test，ABC 契约不变。

1 OrderEnrichmentDataSource 参考实现

billing API 取发票号 → MySQL 富化客户 → reconcile API 对账 POST。一个 execute()，三条 leg，全程 trace。

🏷️ 业务背景 · 声明式引擎表达不了的固定跨类型流程
"登录 billing API → 拉发票 ID → 在 MySQL 查客户详情 → 把每条富化记录 POST 到对账 API。"——固定、每次都一样、且混 HTTP+DB。 声明式编排引擎（仅 HTTP 步骤、无 if/while）表达不了；每次让 Agent 多轮跑又浪费 LLM 成本且依赖多跳可靠性。 答案是 Path 4：用 Python 写一次，作为一个 DataSource。

execute() 代码骨架（组合优于继承 · 强制 trace）

class OrderEnrichmentDataSource(DataSource):
    """Fixed cross-type flow: billing API → MySQL enrichment → reconcile API."""

    def __init__(self, config, http_client, mysql, trace_sink):
        self._config = config
        self._http = http_client
        self._mysql = mysql          # composed DataSource (read-only), NOT inherited
        self._trace = trace_sink

    async def execute(self, query: QueryRequest) -> StructuredResult:
        # root span — kind=orchestration (this IS an orchestrated flow, in code)
        exec_id = await self._trace.start_execution(self._config.id, root_query_id=None)
        root = await self._trace.start_span(exec_id, None, "order_enrichment", "orchestration")
        try:
            # Leg 1 — HTTP: fetch invoice IDs (token injected by TokenManager)
            s1 = await self._trace.start_span(exec_id, root, "billing.fetch_invoices", "http_request")
            invoices = await self._fetch_invoices(query.params)
            await self._trace.finish_span(s1, "ok", {"count": len(invoices)})

            # Leg 2 — SQL: enrich with customer details (read-only, bounded)
            s2 = await self._trace.start_span(exec_id, root, "mysql.enrich_customers", "sql_exec")
            ids = [inv["cust_id"] for inv in invoices]
            enriched = await self._mysql.execute(QueryRequest(   # delegate to composed source
                source_id=self._mysql.id,
                statement="SELECT id, name, tier FROM customers WHERE id IN :ids",
                params={"ids": ids}, timeout=query.timeout))
            await self._trace.finish_span(s2, "ok", {"rows": enriched.row_count})

            # Leg 3 — HTTP: POST each enriched record to reconciliation (foreach, in Python)
            s3 = await self._trace.start_span(exec_id, root, "reconcile.post", "http_request")
            posted = await self._post_reconciliation(enriched.rows, invoices)
            await self._trace.finish_span(s3, "ok", {"posted": posted})

            await self._trace.finish_span(root, "ok", {})
            return StructuredResult(
                columns=[ColumnMeta(name="cust_id", type="BIGINT"),
                         ColumnMeta(name="name", type="VARCHAR"),
                         ColumnMeta(name="reconciled", type="BOOLEAN")],
                rows=posted, row_count=len(posted), execution_time_ms=...)
        except Exception as e:
            # structured error — agent/user never sees a Python traceback
            await self._trace.finish_span(root, "error", {"error": str(e)})
            raise DataSourceError(code="ENRICHMENT_FAILED",
                message=f"Order enrichment flow failed: {e}",
                suggestion="Check billing API health and MySQL connectivity")

对应 Span 树（1 root orchestration + 3 children · 桌面 /history/:id 原生渲染）

httpbilling.fetch_invoices180ms

2 何时封装为自定义源 · 决策树

三条全中才封装；否则走更低层路径。

✅ 封装条件（ALL true）
  • ① 流程固定——每次跑同样的步骤（非 agent 驱动）
  • ② 混源类型——混 HTTP + DB，或需声明式引擎拒绝的逻辑（条件/循环/中流聚合）
  • ③ 高频/关键——重复次数多到 per-run 的 agent 成本/延迟/可靠性是真问题

否则走更低层路径

  • 仅 HTTP、线性/foreach → OrchestratedDataSource（声明式 TOML）
  • 动态、agent 决定、小数据 → Agent 多轮编排（query+params 串联）
  • 既有数据的跨源 JOIN → DuckDB 联邦（CrossSourceDataSource）
不是编排引擎的矛盾：编排引擎保持声明式 HTTP-only（可 review、安全）；自定义源是另一扩展层——新源类型，按代码模块 review/test，恰好在内部混传输。ABC 契约不变，MCP/registry 永不 isinstance 特判。

3 组合优于继承 · Composition over Inheritance

自定义源委托 SQL 给被组合源，不 subclass 它。

🧩 has-a，不是 is-a
OrderEnrichmentDataSource 持有一个 MySQLDataSource（构造注入），在 Leg 2 调 self._mysql.execute(...) 委托 SQL。 它不继承 MySQLDataSource——自定义源是它自己的 DataSource 子类型，SQL 只是它的一条 leg。这让每个源单一职责，可独立 review/test。 
class OrderEnrichmentDataSource(DataSource):      # 继承 ABC，不继承 MySQL
    def __init__(self, config, http_client, mysql, trace_sink):
        self._mysql = mysql                          # ✅ composed (has-a)
        # ❌ class OrderEnrichmentDataSource(MySQLDataSource):  ← 反模式
    async def execute(self, query):
        enriched = await self._mysql.execute(...)   # 委托给被组合源
构造注入：依赖由 registry 从配置解析并注入——与 OrchestratedDataSource 接 config/http_client/token_manager/trace_sink 同模式。被组合源（如 mysql_source_id = "prod-mysql"）按 id 从同一 config 解析。

4 生产护栏 · 自定义源不豁免 红线

自定义源不是法外之地——生产姿态（duckdb-cross-source § Production Posture）每条 execute() 必须满足。

护栏适用 leg说明
Read-only 强制SQL leg只读 DB 账号 + 关键字阻断（SELECT-only）；写操作拒绝
Statement timeout所有 leg累计或 per-leg，但必须有界——挂起的 HTTP leg 不能永远跑
Row capSQL legmax_rows 截断输出，truncated=true 标记
Response size capHTTP leg每条 HTTP leg max_response_bytes 硬上限
结构化错误所有 leg失败抛 DataSourceError{code,message,suggestion}，绝不暴露 traceback（First Principle #5）
TraceSink 强制所有 leg每条 leg 开 span；无 trace 的自定义源在 review 阶段拒绝（黑盒是 audit-trace 要消灭的问题）
失败响亮：任一 leg 失败时，finish_span(..., "error", ...) 标记该 span，已完成的 leg span 保留可见——审计时一目了然哪条 leg 挂了。整体抛结构化错误，agent 拿到 ENRICHMENT_FAILED 而非 Python 崩溃。

5 注册 · type = "custom:<module>" dev-time

自定义源在启动时由 registry 解析——非运行时（区别于内置源的运行时实例化）。

config.toml · 声明自定义源

解析时机：内置类型（mysql/postgresql/http/orchestrated）随 deepferry 发布，运行时由 registry 实例化；自定义类型通过 entry point 或启动时显式 registry 调用注册（M2.5 TBD）。被组合源按 id 从同一 config 解析并注入构造函数——dev-time 扩展，非运行时动态加载。
02

跨源聚合 · 内容场景

DuckDB Federation · v1 核心

"聚合跨源数据 就是 产品存在的理由。" 没有它，deepferry 退化成 N 个独立单源工具。 Agent 发一条 SQL，JOIN/UNION/聚合分散在 MySQL、PostgreSQL、登录态 HTTP API 的数据，拿回一份报表。 DuckDB 嵌入式 in-process federation——不落地仓库，查询时联合。

1 客户 360 视图 经典场景

MySQL 客户基础信息 × 登录态 HTTP API 订单，JOIN 出完整画像。

🏷️ 业务背景 · 单源无法回答
"我们的 Top 客户是谁？"——客户资料在内部 MySQL，但真实订单数据在外部 SaaS 平台（需两步认证）。 分别查两边无法回答这个问题：答案天然需要 JOIN。这正是 BI 工具（Superset/Metabase）存在的理由，deepferry 要为 Agent 驱动的临时分析服务同样的需求。

Agent 发起的跨源 SQL

-- Agent 一条语句，deepferry 用 DuckDB 联合执行
SELECT c.name, c.email, SUM(o.amount) AS total, COUNT(*) AS orders
FROM mysql_src.customers c
JOIN http_orders o ON c.id = o.user_id   # http_orders 已物化为 DuckDB 关系
WHERE c.created_at >= '2024-01-01'     # 谓词下推到 MySQL（红线）
GROUP BY c.name, c.email
ORDER BY total DESC
LIMIT 10;

2 跨系统对账 高频企业场景

内部财务库(PG) × 外部支付平台(HTTP)，FULL OUTER JOIN 找差异。

🏷️ 业务背景 · 找出"对不上的账"
财务系统(PG)记录的退款金额，要与外部支付平台(登录态 API)的实际打款对账。 两边都有，但金额或状态对不上的记录就是问题。FULL OUTER JOIN + 差异过滤，一条 SQL 找出全部异常——过去要导出两张 Excel 手工 VLOOKUP。 
SELECT
  COALESCE(f.order_id, p.order_id) AS order_id,
  f.amount AS internal_refund,
  p.amount AS external_payout,
  CASE WHEN f.amount IS NULL THEN '外部有/内部无'
       WHEN p.amount IS NULL THEN '内部有/外部无'
       WHEN f.amount <> p.amount THEN '金额不符'
  END AS issue
FROM pg_finance.refunds f
FULL OUTER JOIN http_payouts p ON f.order_id = p.order_id
WHERE f.amount IS NULL OR p.amount IS NULL OR f.amount <> p.amount;
价值：跨系统对账是企业数据消费的主导模式。BI 工具正为此而生。deepferry 让 Agent 也能驱动这种临时分析，无需导出手工比对。

3 运营 Top-N 报表

跨源 JOIN + 聚合 + 排序，定时运营分析。

"各城市近 30 天 GMV Top10"——城市/用户在 MySQL，订单在外部 API。 跨源 JOIN + GROUP BY city + ORDER BY gmv DESC。DuckDB 把 SUM/COUNT 下推到 MySQL（aggregate pushdown），JOIN 在本地完成。 
SELECT u.city, SUM(o.amount) AS gmv
FROM mysql_src.users u
JOIN http_orders o ON u.id=o.user_id
GROUP BY u.city ORDER BY gmv DESC LIMIT 10;

4 跨库 UNION 合并

MySQL + PostgreSQL 同构数据合并查询。

历史订单迁移中：新订单在 PostgreSQL，旧订单在 MySQL，schema 一致。 UNION ALL 后按时间统一查询，对应用透明——仿佛一张表。两个 SQL 源都走 ATTACH 实时扫描 + 谓词下推。 
SELECT * FROM mysql_old.orders
WHERE created_at >= '2025-06-01'
UNION ALL
SELECT * FROM pg_new.orders
WHERE created_at >= '2025-06-01';

🛡️ 生产护栏（红线 · 不是优化项）

跨源聚合"看起来危险"的部分被这些硬门挡住。每条都是 init-time 不变量或查询时预检。

① Filter Pushdown 强制

最危险的 footgun：mysql/postgres scanner 默认会全表拉过来再本地过滤。对 1000 万行的生产表做无 WHERE 的 JOIN = 生产事故。deepferry 在启动时强制开启 mysql_enable_predicate_analyzer；若任一 SQL 源 pushdown 关闭/不支持，拒绝启动（PUSHDOWN_DISABLED）。

② EXPLAIN 预检

执行前跑 EXPLAIN，若检测到 attached 源的无谓词全表扫描，拒绝并返回 MISSING_PREDICATE，建议加 WHERE / 时间范围。绝不静默拉全表。

③ 物化三层策略

④ 内存硬边界

memory_limit(默认2G) + temp_directory 溢写磁盘 + max_temp_directory_size(10G)。超 memory 不 OOM 而溢写磁盘（变慢不崩）；超 temp 上限报 TEMP_SIZE_EXCEEDED。engine 未设 memory_limit 则拒绝启动。
Schema 处理：HTTP 资源默认显式声明列（TOML 里写 name/type/nullable），DuckDB 按声明建关系，字段不匹配报错而非静默接受——拒绝采样推断这种不稳定方式（API 会丢可空字段、改 key 顺序）。
移除的"逃生通道"：pandas 回退、无界全量物化、仅采样推断 schema、"30 秒演示"框架——全部删除。失败以结构化错误暴露，绝不掩盖。这是 v1 生产姿态，不是 demo。

🔄 热重载交互 · registry.refresh() 对三类源一视同仁 Hot Reload

config-panel CRUD 写入后，下一次 query() 即生效——无需进程重启。Copy-on-Write 指针 swap 对所有源类型安全。

① COW 指针 swap · 全类型安全

refresh() 构建新 instances dict，在 GIL 下原子替换 registry 内部指针；query() 始终读当前指针。在途查询持有它启动时的源实例引用，Python GC 保活至返回——对编排源、跨源、自定义源同样安全，无需类型特判。

② 自定义源实例 drain · 同规则

自定义源实例的 drain 与任何源一致：removed/edited 进入 draining 状态，后台 asyncio 任务在优雅期（默认 30s，≥ statement timeout）后关闭；引用全落则立即关闭。其内部组合的 MySQLDataSource 随之 drain——组合关系不延长生命周期。

③ 跨源 JOIN 的 ATTACH/DETACH 串行化

写锁串行：当参与跨源联邦的源（含自定义源）被增删时，对应 ATTACH/DETACH 在 DuckDB writer lock 下串行执行（ATTACH 本身是串行操作）。已在飞的跨源查询持有自己的 DuckDB 连接与其 attached 源快照——不受 refresh 影响。

④ 审计

每次 refresh() 记录变更（added/edited/removed 源 id、触发者）到 trace store，凭证字段按 audit-trace 规约脱敏——自定义源的 dev-time 注册信息同样可审计。
三类源共享生命周期：编排源、自定义源、直连源在 registry 里都是 DataSource 实例——同一 refresh()、同一 COW swap、同一 drain、同一 ATTACH 串行规则。能力分层在"如何拿数据"，不在"如何被管理"。
04

能力边界与设计原则

明确"不做什么"同样重要

📐 共同的设计原则

  • DataSource ABC 神圣不可破：编排是子类型，不是 MCP 之上的层；永不 isinstance 特判
  • 声明式配置：流程/资源都在 config.toml，git-diffable，可 review；无用户 Python
  • 错误结构化：所有失败是 JSON {code, message, suggestion}，绝不暴露 traceback
  • Agent 简单：始终一次 query() / 一条 SQL → 一个 StructuredResult；复杂度藏在数据源内
  • 失败响亮：pushdown 关闭、全表扫描、超内存——拒绝而非掩盖
  • 审计非可选：每次执行产 span 树，OTel 对齐，可导出 OTLP

🚫 明确不做（Out of Scope）

  • 编排：条件 if、循环 while、任意 DAG、Jinja 过滤器、用户 Python
  • 认证：无限重试退避（仅单次 401 重试）、OAuth2 完整授权码流
  • 跨源：无界全量物化、pandas 回退、采样推断 schema
  • 引擎：Trino/Presto 这类重型联邦服务器（DuckDB 单机嵌入式已足够）
  • 规模：RBAC/多租户、Kubernetes/Helm（单机部署定位）
  • 方向：这些限制是刻意的——少而精的生产功能 > 多而糙的 demo
