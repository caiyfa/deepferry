# Capability: Explore Mode (NL → SQL)

> **Status**: planned | **Milestone**: M3 | **Owner**: frontend + backend | **Depends on**: `console-shell`, `llm-integration`

## Summary

A natural-language exploration mode where users type questions in plain
language and deepferry generates SQL, executes it, and returns results with
auto-generated charts. Zero SQL knowledge required. Power users can inspect
or modify the generated SQL at any time.

## Motivation

- The current Query page requires users to write SQL manually. Business
  stakeholders and non-technical users cannot use deepferry at all.
- deepferry's value is "universal data access" — gatekeeping that behind SQL
  literacy contradicts the product vision.
- The `openai>=2.43` dependency already exists in `pyproject.toml` (noted
  as "DeepSeek via base_url") but has never been used. This spec activates it.
- NL→SQL is the most requested capability in data tools (evidenced by
  Vanna.ai, SQLChat, Databricks AI, etc.). It is table-stakes for a modern
  data console.

## Specification

### Interaction Flow

```
[Empty State] → [Type NL question] → [LLM generates SQL] → [Execute] → [Show results + charts] → [Follow-up question]
```

### Empty State

- Centered greeting: "用自然语言问你的数据"
- 3-4 suggested questions generated from the selected data source(s) schema.
  When multiple sources are selected, suggestions bias toward cross-source
  questions (e.g., "关联 MySQL 客户表和 API 发票数据")
- Schema preview panel below: lists tables and row counts for currently
  selected source(s)

### Question Input

- Multi-line textarea, Enter to submit, Shift+Enter for newline
- Placeholder adapts to single vs multi-source mode
- Submitting triggers the LLM pipeline

### Progress Indication (Streaming)

The LLM pipeline is opaque unless progress is shown. Each step emits a status
event via SSE:

```
✓ 识别数据源: mysql-main, finance-api
✓ 匹配表: customers + /api/v1/invoices
⟳ DuckDB 编排跨源 JOIN...
⟳ 执行查询...
```

### Result Display

- Natural-language summary line (generated by LLM or template)
- Data table (AG Grid, same component as Query mode)
- Auto-generated charts:
  - Numeric columns → line/bar chart (time-series if date column detected)
  - Categorical columns → pie/donut chart
  - Single numeric value → big-number card
- Collapsible SQL panel showing the exact generated SQL
- Action bar: Copy data, Export CSV, Save as Dataset (→ `dataset-ui`), Bookmark

### Follow-up Questions

- Input field below results, preserved in conversation history
- Context from previous question/result passed to LLM for referential understanding
  (e.g., "这些产品的退货率" → LLM knows "这些产品" refers to the 3 products
  in the previous result)

### Error Handling

When LLM cannot generate valid SQL:
- Show the attempted interpretation
- Offer suggestions: try a different phrasing, browse the schema, or switch
  to Query mode
- Never show raw Python tracebacks to the user

When LLM is unavailable:
- Explore mode falls back to schema browser + suggested questions
- Show a banner: "AI 服务暂不可用，请使用查询模式或浏览数据源"

### Multi-source Mode

When 2+ data sources are selected:
- LLM prompt includes schema from all sources
- Generated SQL uses DuckDB cross-source syntax (`mysql_main.customers JOIN http_finance.invoices`)
- Result display includes an orchestration flow diagram (→ `cross-source-ui` spec)
- Performance: each source's query time shown in a mini-timeline

### Configuration

```toml
[llm]
provider = "deepseek"           # or "openai", "local"
api_key = "${DEEPFERRY_LLM_API_KEY}"
model = "deepseek-chat"         # or "gpt-4o", etc.
base_url = "https://api.deepseek.com/v1"  # optional, for compatible APIs
max_tokens = 2000
temperature = 0.1               # low for deterministic SQL generation
```

### API Endpoints

| Endpoint | Method | Request | Response |
|---|---|---|---|
| `/api/explore` | POST | `{question, source_ids, conversation_history?}` | SSE stream of progress + final `StructuredResult` |
| `/api/explore/suggestions` | GET | `?source_ids=mysql-main,finance-api` | `{suggestions: string[]}` |

### Prompt Engineering

The LLM prompt includes:
1. System role: "You are a SQL expert for deepferry..."
2. Schema context: table names + columns + types + row counts + sample values
3. Few-shot examples: 3 question→SQL pairs matching the schema
4. Cross-source hints: when multiple sources, note which columns can be joined
5. Safety rules: SELECT only, no DDL/DML, must include LIMIT unless aggregation
6. Output format: `{sql, explanation}` JSON

### Safety

- LLM-generated SQL is re-scanned by the existing `_scan_sql` safeguard before execution
- `max_rows` enforced regardless of LLM output
- All queries are `SET SESSION TRANSACTION READ ONLY` (existing behavior)

## Acceptance Criteria

- [ ] Type "上个季度销售额最高的三个产品" → table shows 3 products with revenue
- [ ] Type "查看不存在的客户满意度" → friendly error with schema browsing suggestion
- [ ] Follow-up "这些产品的退货率" → context preserved, correct query generated
- [ ] Select mysql-main + finance-api → type "关联客户和发票" → cross-source result
- [ ] Set `DEEPFERRY_LLM_API_KEY` to empty → explore mode shows graceful fallback
- [ ] Generated SQL panel is collapsible; modifying and re-executing updates results
- [ ] `config.toml` change `[llm].provider = "openai"` works without code change
- [ ] Auto-chart appears for numeric columns; no crash on all-text results

## Out of Scope

- Multi-turn conversational memory beyond the current session (not a chatbot)
- NL → chart specification ("画个饼图") — auto-chart only, no user-directed visualization
- Training/fine-tuning models — zero-shot + few-shot prompt only
- Voice input
