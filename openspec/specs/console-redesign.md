## 1. 产品定位

### 1.1 核心问题

当前前端是一个 **SQL 客户端**：选数据源 → 手写 SQL → 看表格。在 AI Agent 时代，这个设计有根本性缺陷：

- **Agent 开发者** 看不到 agent 在调什么 query，无法调试和优化
- **业务人员** 被迫手写 SQL，工具形同虚设
- **所有人都要懂 schema** 才能用，探索成本极高
- **跨源能力被隐藏**：deepferry 最强的 DuckDB 多源 JOIN 能力在前端完全不可见

### 1.1b deepferry 独有的异构数据源能力

deepferry 不是又一个 SQL 客户端，它的真正壁垒是：

```
Agent 的自然语言
     ↓
deepferry 编排引擎
 ├── MySQL    →  SELECT * FROM customers     (7 rows, 156ms)
 ├── HTTP API →  GET /api/v1/invoices         (8 rows, 250ms)
 └── DuckDB   →  JOIN ON tax_no              (5 rows, 62ms)
     ↓
统一结果 + 可视化
```

当前前端只暴露了「选一个源 → 写 SQL」的单源场景，而 deepferry 的编排能力（多源并行查询 + DuckDB 内存 JOIN）完全没有在前端呈现。

### 1.2 目标用户双画像

| | Agent 开发者 | 业务人员 |
|---|---|---|
| **场景** | 调试 agent 生成的 SQL、监控 agent session、追踪执行链路 | 用自然语言问数据问题、看趋势、导出报告 |
| **技能** | 懂 SQL | 不懂 SQL |
| **频率** | 日常高频 | 周期性（周报/月报） |
| **核心诉求** | "agent 又写错了什么 query，为什么慢" | "上个季度哪个产品卖得最好" |

### 1.3 设计原则

1. **Agent-first**：默认视图是 agent 在做什么，不是 SQL 编辑器
2. **零 SQL 可用**：探索模式完全不需要 SQL 知识
3. **SQL 透明化**：高级用户随时可以看到/修改生成的 SQL
4. **跨源可视化**：数据编排流程（多源并行 → DuckDB JOIN）全程可视化
5. **即时反馈**：所有操作 < 500ms 内响应（乐观更新 + 流式）
6. **数据源抽象**：不管底层是 MySQL、PG、HTTP API 还是 DuckDB 组合，界面表现一致

| 交互 | 说明 |
|---|---|
| 自然语言输入 | 支持中文，自动识别意图（查询/统计/对比/趋势） |
| 动态推荐 | 基于所选数据源的表和列自动生成推荐问题 |
| 流式反馈 | AI 分析过程逐步展示（识别源→匹配表→生成 SQL→执行） |

| 多轮追问 | 保持上下文，支持"这些产品中"、"退换货情况"等指代 |
| 自动图表 | 数值列自动生成趋势图，分类列自动生成分布图 |
| 数据源预览 | 输入框下方可浏览表结构，点击表名预览前 5 行 |

## 10. 架构决策：deepferry 是否需要自己的 LLM

### 10.1 问题

探索模式（NL → SQL）意味着 deepferry 需要语言模型能力。这是否违背了 deepferry 作为纯 MCP 数据访问层的定位？

### 10.2 结论：需要，但不侵入 MCP 层

deepferry 需要自己的 LLM，但应该只加在 **Sidecar API 层**，不碰 MCP 协议。

```
┌─────────────────────────────────────────────────────────┐
│                      deepferry                           │
│                                                         │
│  MCP Protocol (Agent ↔ Tool)    Sidecar API (Human ↔ UI)│
│  ═══════════════════════════     ═══════════════════════ │
│  list_sources                    GET  /api/config        │
│  execute_query                   POST /api/query         │
│  schema_info                     POST /api/explore  ← NEW│
│  cross_query                          │                  │
│  propose_source                   ┌───┴──────────┐       │
│                                   │  LLM Client   │       │
│  不变：纯数据工具，               │  NL → SQL     │       │
│  Agent 专用                       │  openai SDK   │       │
│                                   │  ↑已有依赖    │       │
│                                   └──────────────┘       │
└─────────────────────────────────────────────────────────┘
```

### 10.3 为什么不复用 Agent 的 LLM

| 方案 | 问题 |
|---|---|
| UI → Agent → deepferry（反向 MCP） | 循环依赖、链路太长、要求用户始终连着 Agent |
| MCP sampling 协议 | 依赖客户端提供 LLM，非所有客户端支持，deepferry 无法独立工作 |
| **deepferry 自带 LLM** ✅ | 独立产品、直接可控、延迟低、无需外部依赖 |

### 10.4 行业验证

已有多个 MCP 服务器内置 LLM 用于 NL→SQL：
- Vanna.ai MCP（内置 LLM 生成 SQL）
- SQLChat MCP（chat 接口 + LLM backend）
- 多个 text-to-SQL MCP 实现

这些案例证明：**MCP 服务器嵌入 LLM 能力是常见的、被验证的模式**。

### 10.5 实施要点

| 关注点 | 方案 |
|---|---|
| LLM 后端 | 可配置（DeepSeek / OpenAI / 本地模型），`config.toml` 的 `[llm]` section |
| API 密钥 | 环境变量 `${DEEPFERRY_LLM_API_KEY}`，不写进 config |
| NL→SQL 流程 | `/api/explore`：接收 NL → 构造 prompt（schema context） → LLM 生成 SQL → 安全检查 → 执行 |
| MCP 层不变 | Agent 依然通过 MCP 工具调用 deepferry，探索模式是给人类用的 sidecar |
| 已有依赖 | `pyproject.toml` 中 `openai>=2.43` 已规划，直接复用于 LLM client |
| 降级策略 | LLM 不可用时，探索模式回退为 Schema 浏览 + 示例问题，不阻断使用 |

```python
# duckdb.py L302-323: _build_insert_values
# HTTP JSON → Python dict → str() 强转 → SQL INSERT VALUES → DuckDB 表
# 嵌套对象丢失、类型信息丧失、数据锁死在 DuckDB 专有格式
```

每次查询: ATTACH MySQL (live) → 重拉 HTTP API → `str()` 拍平 → `INSERT VALUES` → DuckDB `:memory:` → 返回 → 丢弃

三个问题：
1. **重复拉取**：HTTP API 每次重新请求
2. **无法回溯**：上次查询结果已消失
3. **格式锁死**：数据一进 DuckDB 就出不来，其他工具无法使用

### 11.2 设计原则

**存储与查询分离。** DuckDB 不负责存储，只负责查询。数据以开放格式落盘，任何工具都能读取。

```
设计原则:
  存储层 = 格式无关（JSON / Parquet / Arrow / CSV）
  查询层 = DuckDB 原生读取（read_json / read_parquet / read_csv）
  回退   = 原始文件在任何工具中都能直接用
```

### 11.4 格式选择策略

| 数据来源 | 存储格式 | 理由 |
|---|---|---|
| MySQL/PostgreSQL 查询结果 | **Parquet** | 列式、压缩高、类型严格、所有数据分析工具都能读 |
| HTTP API JSON 响应 | **JSON** (原始) | 保留嵌套结构、API 响应原样可审计 |
| 跨源 JOIN 结果 | **Arrow IPC** | 零拷贝、DuckDB/Pandas/Polars 共用、内存映射 |
| CSV 文件导入 | **CSV** (原始) + Parquet (索引) | 保留原始文件，Parquet 加速查询 |
| 大数据集 (>100K rows) | **Parquet (按列分区)** | 只读需要的列，减少 IO |

**为什么不只用一种格式？**
- JSON 保留嵌套结构，但查询慢（需 parse）
- Parquet 查询快，但扁平化会丢掉嵌套
- Arrow 零拷贝，但不是所有工具支持
- **不同来源用最适合的格式，DuckDB 统一查询**

### 11.5 DuckDB 角色转变

```python
# 旧设计（格式锁死）
conn = duckdb.connect("datasets/my_ds.duckdb")  # 数据困在 DuckDB 里
conn.execute("INSERT INTO t VALUES (...)")       # 手拼 SQL 插入

# 新设计（格式无关）
conn = duckdb.connect(":memory:")  # 或缓存文件
conn.execute("CREATE VIEW invoices  AS SELECT * FROM read_json('v1/invoices.json')")
conn.execute("CREATE VIEW customers AS SELECT * FROM read_parquet('v1/customers.parquet')")
conn.execute("SELECT * FROM customers JOIN invoices ON ...")
# DuckDB 只做查询引擎，原始文件不受影响
```

**DuckDB 的角色：** 从"存储引擎"降级为"查询加速层"。`_duckdb_cache/` 是可重建的，删了不影响数据完整性。

### 11.6 版本管理模型

| 概念 | 说明 | 存储 |
|---|---|---|
| **Dataset** | 一个逻辑数据集（命名、描述、源查询） | `metadata.yaml` |
| **Snapshot** | 一个时间点的数据快照（多个文件） | `v1/` 目录 |
| **Version** | 用户可见的版本号（v1, v2, v3） | `metadata.yaml` 中的版本链 |
| **Diff** | 两个快照的差异 | 通过 DuckDB `EXCEPT` / `INTERSECT` 计算 |
| **Fingerprint** | 数据完整性校验 | `manifest.yaml` 中的 SHA256 |

**版本链：**
```
v1 ────→ v2 ────→ v3
2026-06   2026-07   2026-08
 │         │         │
 7 rows    9 rows    8 rows
           +2        -1 (客户流失)
```

### 11.7 增量更新

不是每次刷新都全量拉取：

```python
# HTTP API 增量: 只拉更新的数据
GET /api/v1/invoices?updated_since=2026-06-30T10:00:00

# MySQL 增量: WHERE 过滤
SELECT * FROM customers WHERE updated_at > '2026-06-30 10:00:00'

# 增量策略配置
dataset:
  refresh_strategy: incremental  # or full
  incremental_column: updated_at  # 用于 WHERE 的时间戳列
```

### 11.10 存储估算

| 数据规模 | Parquet (压缩) | JSON (原始) | Arrow |
|---|---|---|---|
| 10 列 × 10K 行 | ~100 KB | ~1.5 MB | ~800 KB |
| 20 列 × 100K 行 | ~2 MB | ~30 MB | ~15 MB |
| 50 列 × 1M 行 | ~40 MB | ~300 MB | ~200 MB |

**100 个数据集 × 10 版本 at 10K rows ≈ 100 MB**（Parquet）或 **1.5 GB**（JSON）
